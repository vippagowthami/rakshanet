## Deployment Options

Option A — Deploy on Render (recommended, builds from Dockerfile):

1. Create an account at https://render.com and connect your GitHub repository.
2. Create a new Web Service → select your repository and set the following:
   - Environment: Docker
   - Build Command: leave default (Render builds the Dockerfile)
   - Start Command: leave empty (Dockerfile CMD is used)
3. Add the following environment variables in your Render service settings:
   - `SECRET_KEY` — a secure Django secret key
   - `DEBUG` — `False` for production
   - `ALLOWED_HOSTS` — `.onrender.com,localhost,127.0.0.1`
   - `CSRF_TRUSTED_ORIGINS` — `https://*.onrender.com,http://localhost:8000`
   - Email settings if you need email notifications
4. Deploy — Render will build the image and provide a public URL.
5. **Enable Auto-Deploy:** In Render dashboard → Settings → Auto-Deploy from Git (select `main` branch)

Option B — Deploy to Heroku (container mode):

1. Install the Heroku CLI and login: `heroku container:login`.
2. Create an app: `heroku create <app-name>`.
3. Build and push the container:
   ```bash
   docker build -t registry.heroku.com/<app-name>/web .
   docker push registry.heroku.com/<app-name>/web
   heroku container:release web -a <app-name>
   ```
4. Set config vars in Heroku (`SECRET_KEY`, `DEBUG=False`, email settings).

Option C — GitHub Actions + GitHub Packages (CI build + publish):

- See `.github/workflows/docker-publish.yml` for a starter workflow that builds and publishes a Docker image to GitHub Container Registry. You will need to set `CR_PAT` (a personal access token) in your repo secrets.

## Continuous Deployment Setup

### How It Works
Every time you push to the `main` branch, Render automatically:
1. Pulls the latest code from GitHub
2. Builds a new Docker image
3. Runs database migrations
4. Collects static files
5. Starts the new service
6. Routes traffic to the new deployment

### Push to Deploy
```bash
# Make changes locally
git add .
git commit -m "Your changes"

# Push to GitHub
git push origin main

# Render automatically detects the push and deploys within 2-3 minutes
# Check deployment status at: https://dashboard.render.com/
```

### Deployment Logs
- View deployment logs in Render Dashboard: https://dashboard.render.com/
- Check build and runtime logs in real-time
- Monitor for errors or warnings

### Rollback
If deployment fails:
1. Go to Render Dashboard → RakshaNet Service → Deployments
2. Select a previous successful deployment
3. Click "Redeploy" to rollback

## Environment Variables

Production environment variables are configured in Render Dashboard:

```
SECRET_KEY=<auto-generated secure key>
DEBUG=False
ALLOWED_HOSTS=.onrender.com,localhost,127.0.0.1
CSRF_TRUSTED_ORIGINS=https://*.onrender.com,http://localhost:8000,http://127.0.0.1:8000
CSRF_COOKIE_SECURE=True
SESSION_COOKIE_SECURE=True
SECURE_SSL_REDIRECT=True
SECURE_HSTS_SECONDS=31536000

# Database
DATABASE_URL=<PostgreSQL connection string>

# Email (optional)
EMAIL_HOST=smtp.gmail.com
EMAIL_PORT=587
EMAIL_HOST_USER=<your-email>
EMAIL_HOST_PASSWORD=<your-app-password>
```

## Docker Build & Run Locally

```bash
# Build Docker image
docker build -t rakshanet:latest .

# Run Docker container
docker run -p 8000:8000 \
  -e SECRET_KEY=your-secret-key \
  -e DEBUG=True \
  -e ALLOWED_HOSTS=localhost,127.0.0.1 \
  rakshanet:latest
```

## Performance Optimization

Current Render free-tier setup:
- **CPU:** Shared (0.5 Compute Units)
- **RAM:** 512 MB
- **Database:** PostgreSQL
- **Auto-spinning:** Sleeps after 15 minutes of inactivity (free tier only)

For production scaling:
- Upgrade to **Starter or Standard plan** for dedicated CPU
- Use **PostgreSQL Pro** for better database performance
- Enable **Redis** for caching (optional)
- Configure **CDN** for static file serving (optional)
